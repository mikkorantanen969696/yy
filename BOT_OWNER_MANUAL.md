# Инструкция для будущего владельца бота

## 1. Что находится в проекте

- `app/main.py` - точка входа (`python -m app.main`), запуск в режимах polling/webhook.
- `app/config/settings.py` - чтение настроек из `.env`.
- `app/db/` - подключение к БД и авто-инициализация схемы.
- `app/models/` - таблицы БД (`users`, `orders`, `responses`, `role_invites`, фото и т.д.).
- `app/services/` - бизнес-логика (пользователи, заявки, аналитика, экспорты, инвайты).
- `app/handlers/` - Telegram-команды и сценарии ролей (admin/manager/master/common).
- `app/utils/` - константы, клавиатуры, форматирование текста.
- `tests/`, `scripts/` - заготовленные директории (сейчас пустые).
- `DEPLOY.md` - краткий чек-лист деплоя.
- `.env.example` - шаблон переменных окружения.
- `.env` - рабочие настройки (секреты, ID, режим запуска).

## 2. Роли и доступы

- `admin` - полный доступ к панели и управлению.
- `manager` - создание и сопровождение заявок.
- `master` - принятие работ, загрузка фото ДО/ПОСЛЕ, завершение.

Как дается роль:
- Автоматически по `ADMIN_IDS` из `.env` (для владельца/админов).
- Через одноразовое секретное слово, привязанное к `@username`.
- Через команду админа `/set_role`.

Важно:
- Теперь назначение ролей и мастера поддерживает не только `telegram_id`, но и `@username`.
- Назначение по `@username` работает только если пользователь уже есть в БД (он должен хотя бы один раз открыть бота и нажать `/start`).

## 3. Ключевые команды

### Админ
- `/admin` - панель.
- `/stats`, `/city_stats` - аналитика.
- `/orders`, `/order [id]` - список и карточка заявки.
- `/set_status [order_id] [status]` - смена статуса.
- `/reassign [order_id] [master_tg_id|@username|none]` - назначить/снять мастера.
- `/users [role|all] [active|inactive|all] [limit]` - список пользователей.
- `/set_role [telegram_id|@username] [admin|manager|master]` - назначить роль.
- `/set_active [telegram_id] [on|off]` - включить/выключить доступ.
- `/broadcast [role|all] [текст]` - рассылка.
- `/export_basic`, `/export_full` - выгрузки CSV.

### Менеджер
- `/manager`
- `/new_order`
- `/my_orders`
- `/my_stats`
- `/my_export_basic`

### Мастер
- `/profile`
- `/my_jobs`
- `/my_stats`

### Общие
- `/start`
- `/help`
- `/owner_guide`

## 4. Настройки `.env` (что обязательно проверять)

Минимум перед запуском:
- `BOT_TOKEN`
- `DATABASE_URL`
- `RUN_MODE`
- `ADMIN_IDS`
- `GROUP_CHAT_ID`
- `CITY_TOPIC_*` (thread id по каждому городу)

Для webhook:
- `WEBHOOK_URL`
- `WEBHOOK_PATH`
- `APP_HOST`
- `APP_PORT`

Для Postgres SSL при необходимости:
- `DB_SSL_MODE`
- `DB_SSL_ROOT_CERT`

## 5. Первый запуск на новом сервере

1. Создать виртуальное окружение.
2. Установить зависимости: `pip install -r requirements.txt`.
3. Заполнить `.env` на основе `.env.example`.
4. Запустить: `python -m app.main`.
5. Проверить в Telegram:
   - `/start` у владельца.
   - `/admin` открывается.
   - Создание тестовой заявки через `/new_order`.
   - Отклик мастера и завершение с фото.

## 6. Как безопасно передать проект другому владельцу

1. Перевыпустить `BOT_TOKEN` в BotFather перед передачей.
2. Передать новый токен только через защищенный канал.
3. Удалить старые/лишние `ADMIN_IDS` из `.env`.
4. Добавить нового владельца в `ADMIN_IDS`.
5. Проверить права бота в Telegram-группе и топиках.

## 7. Операционные процедуры (ежедневно/еженедельно)

Ежедневно:
- Проверять `/stats` и `/orders`.
- Проверять ошибки в логах процесса.

Еженедельно:
- Делать бэкап БД (`data.db` для SQLite или dump Postgres).
- Проверять `/users` и отключать неактуальных (`/set_active ... off`).
- Проверять корректность городов и topic id.

## 9. Частые проблемы и быстрые решения

Проблема: `@username` не находится при `/set_role` или `/reassign`.
- Причина: пользователь еще не зарегистрирован в локальной БД.
- Решение: попросить пользователя открыть бота и выполнить `/start`, затем повторить команду.

Проблема: заявки не публикуются в группу.
- Проверить `GROUP_CHAT_ID`, `CITY_TOPIC_*`, права бота в группе.

Проблема: "Нет доступа" у сотрудника.
- Проверить роль пользователя (`/users`).
- Проверить активность пользователя (`/set_active`).
- Проверить, что у сотрудника корректный `@username` (для входа по секретному слову).
